# npm audit report (2026-06-09)

Scope:

- `frontend/package-lock.json`
- `vscode/package-lock.json`

Method:

- Ran `npm audit --json` against each lockfile on 2026-06-09.
- Verified flagged packages against the committed manifests and lockfiles.

## Findings

### frontend

Direct dependency:

- `svelte` declared as `^5.19.0` in `frontend/package.json`

Locked packages:

- `svelte@5.55.0`
- `devalue@5.6.4`

Advisories reported:

- `svelte` moderate:
  - GHSA-f3cj-j4f6-wq85
  - GHSA-rcqx-6q8c-2c42
  - GHSA-9rmh-mm8f-r9h6
  - GHSA-pr6f-5x2q-rwfp
- `devalue` high:
  - GHSA-77vg-94rm-hx3p

Assessment:

- Both packages are dev dependencies in this repo.
- The frontend uses Vite with the Svelte plugin and does not include SvelteKit or an SSR configuration.
- That reduces the practical exposure for the SSR/XSS advisories, but the vulnerable compiler/runtime versions are still present in the build toolchain and should be upgraded.

Recommended action:

- Update `svelte` to a version newer than `5.55.6`.
- Refresh the lockfile so `devalue` resolves above `5.8.0`.
- Re-run `npm audit` after the upgrade.

### vscode

Direct dependency:

- `mocha` declared as `^10.7.3` in `vscode/package.json`

Locked packages:

- `mocha@10.8.2`
- `serialize-javascript@6.0.2`

Advisories reported:

- `serialize-javascript` high:
  - GHSA-5c6j-r48x-rmvq
  - GHSA-qj8w-gfj5-8c6v
- `mocha` moderate because it pulls the vulnerable `serialize-javascript`

Assessment:

- Both packages are dev dependencies used by the VS Code extension test harness.
- The vulnerable path is not part of the shipped extension runtime, but it is still present in local/CI test execution.
- `npm audit` proposes `mocha@8.1.3`, which is a downgrade and not an acceptable remediation. The correct fix is to move to a Mocha release that pulls a patched `serialize-javascript`.

Recommended action:

- Upgrade `mocha` to a release that resolves `serialize-javascript` above `7.0.4`, or override that transitive dependency if the chosen Mocha version supports it cleanly.
- Re-run `npm audit` after the lockfile changes.

## Summary

- `frontend`: 2 findings (`1 high`, `1 moderate`), both in dev dependencies.
- `vscode`: 2 findings (`1 high`, `1 moderate`), both in dev dependencies.
- No production/runtime npm vulnerabilities were identified in the current manifests, but the build/test toolchains are carrying known vulnerable versions.
